# ADR 0001: Использование SignalR с Redis backplane для горизонтального масштабирования

## Метаданные

| Поле | Значение |
|------|----------|
| **Статус** | Принято (Accepted) |
| **Дата** | 2026-04-24 |
| **Авторы** | Команда / владелец репозитория |

---

## Контекст (Context)

Демонстрационный проект **highload-realtime-signalr-demo** нацелен на показ способности стека .NET обрабатывать высокую real-time нагрузку: множество одновременных подключений и высокий поток сообщений. Планируется запуск **нескольких инстансов** приложения за балансировщиком.

SignalR по умолчанию хранит состояние соединений и групп **в памяти процесса**. При scale-out клиенты, подключённые к разным узлам, не получают общие broadcast-сообщения без **внешней шины**, синхронизирующей события между инстансами.

Требования:

- горизонтальное масштабирование без потери семантики «все подписчики группы получили сообщение»;
- воспроизводимое локально и в CI окружение для интервью и портфолио;
- предсказуемая модель стоимости и эксплуатации для **демо**, не обязательно enterprise multi-region.

---

## Решение (Decision)

Использовать **ASP.NET Core SignalR** с **Redis в качестве backplane** для распространения сообщений между инстансами приложения.

Интеграция ожидается через стандартный механизм `AddStackExchangeRedis` (или актуальный эквивалент в документации Microsoft для используемой версии пакета) при конфигурации SignalR в `Program.cs` / `Startup`.

---

## Последствия (Consequences)

### Положительные

- Официально поддерживаемый паттерн scale-out для SignalR.
- Низкая задержка pub/sub Redis хорошо подходит для fan-out сообщений между узлами.
- Простой подъём Redis в Docker для демо и нагрузочных тестов.
- Знакомый стек для .NET-интервьюеров и ревьюеров.

### Отрицательные / риски

- Redis становится **критической зависимостью** для межузловой доставки; падение Redis нарушает согласованность broadcast между репликами (локальные соединения на узле могут продолжать работать частично).
- Нужно мониторить Redis: задержки, память, blocked clients, сетевые лимиты.
- Сообщения проходят через дополнительный hop — при экстремально больших payload или очень высокой частоте нужен профиль и возможная оптимизация (размер сообщений, сегментация групп).

---

## Альтернативы (Alternatives)

| Альтернатива | Почему не выбрана как основная для демо |
|--------------|----------------------------------------|
| **Azure SignalR Service** | Управляемый сервис сильно упрощает scale-out, но завязан на облако и учётную запись; хуже для «запустил у себя в Docker» на интервью. |
| **SQL Server backplane** (если доступен для версии) | Выше задержка и нагрузка на БД по сравнению с Redis для чистого pub/sub fan-out. |
| **Собственная шина на Kafka / NATS** | Гибко для больших систем, но избыточно по сложности для целей портфолио-демо и не является стандартным «из коробки» путём SignalR. |
| **Один инстанс без backplane** | Проще, но не демонстрирует горизонтальное масштабирование real-time. |

При эволюции проекта возможен **дополнительный** ADR о гибриде (например, Redis для демо + документированный путь миграции на Azure SignalR для продакшн-аналога).

---

## Статус реализации

На момент создания ADR код backplane в репозитории может быть **ещё не добавлен**; ADR фиксирует направление до реализации, чтобы контрибьюторы и интервьюеры понимали принятое архитектурное решение.
