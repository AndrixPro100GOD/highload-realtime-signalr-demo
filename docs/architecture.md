# Архитектура проекта

Документ описывает **целевую** архитектуру демонстрационного high-load решения на .NET + SignalR и **текущий** состав репозитория. Цель — масштабируемый real-time бэкенд, наблюдаемость и горизонтальное масштабирование.

## Цели и ограничения

| Цель | Описание |
|------|----------|
| **Throughput** | Десятки тысяч сообщений в секунду при распределённой публикации |
| **Concurrency** | Десятки тысяч одновременных WebSocket-подключений (SignalR) |
| **Scale-out** | Несколько инстансов приложения за балансировщиком без «тихих» комнат |
| **Observability** | Метрики, трассировки, логи для анализа узких мест под нагрузкой |

Ограничения демо-проекта: упрощённая доменная модель, фокус на инфраструктурных паттернах, а не на бизнес-логике продакшн-системы.

## Текущее состояние репозитория

На момент написания документа в решении присутствует **клиентское** приложение:

- **Blazor WebAssembly** (.NET 10) — UI и точка входа для будущей интеграции с SignalR-клиентом.
- **MudBlazor** — компоненты интерфейса.

Отдельный **ASP.NET Core** хост с **SignalR Hub**, **Redis backplane** и контейнеризацией планируется как следующий этап развития репозитория (см. [ADR 0001](../adr/0001-use-signalr-with-redis-backplane.md)).

## Целевая логическая архитектура

```text
                    ┌─────────────────┐
                    │ L7 Load Balancer│
                    │ (sticky sessions │
                    │  или cookie)    │
                    └────────┬────────┘
                             │
         ┌───────────────────┼───────────────────┐
         ▼                   ▼                   ▼
   ┌───────────┐       ┌───────────┐       ┌───────────┐
   │ App Pod 1 │       │ App Pod 2 │       │ App Pod N │
   │ SignalR   │       │ SignalR   │       │ SignalR   │
   └─────┬─────┘       └─────┬─────┘       └─────┬─────┘
         │                   │                   │
         └───────────────────┼───────────────────┘
                             │
                    ┌────────▼────────┐
                    │ Redis backplane │
                    │ (pub/sub шина   │
                    │  между узлами)  │
                    └─────────────────┘
```

Компоненты:

| Компонент | Роль |
|-----------|------|
| **Клиент (Blazor / SPA / консольный load-gen)** | Подключение к Hub, группы, получение broadcast |
| **ASP.NET Core + SignalR** | WebSocket / Long Polling, авторизация соединений, группы |
| **Redis backplane** | Доставка сообщений между инстансами при scale-out |
| **Балансировщик** | Распределение новых соединений; важна согласованность с sticky-поведением при необходимости |

## Потоки данных (real-time)

1. Клиент устанавливает соединение с одним из инстансов Hub.
2. Сообщение публикуется в группу или всем клиентам на **этом** узле — доставка локально.
3. Через **backplane** событие реплицируется на остальные узлы; там SignalR доставляет его своим подписчикам.

Так обеспечивается **согласованный broadcast** при горизонтальном масштабировании без общей in-memory шины на одном сервере.

---

## Масштабирование

- **Вертикально:** больше CPU/RAM на инстанс, лимиты Kestrel и ОС на открытые сокеты.
- **Горизонтально:** несколько реплик приложения + Redis backplane (см. ADR 0001).
- **Сеть:** таймауты прокси, лимиты соединений, keep-alive — учитывать на границе L7.

Рекомендуется нагрузочное тестирование с реалистичным числом соединений и паттерном сообщений (см. [performance.md](./performance.md)).

## Наблюдаемость (observability)

Целевой минимум для демо high-load:

| Слой | Примеры |
|------|---------|
| **Метрики** | RPS Hub, активные соединения, задержки, ошибки Redis |
| **Логи** | Structured logging, correlation id |
| **Трассировки** | OpenTelemetry: вход HTTP → Hub → вызовы Redis |

Это позволяет на интервью и в портфолио показать не только «много соединений», но и **где** возникает bottleneck.
